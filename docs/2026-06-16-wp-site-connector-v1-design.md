# WP Site Connector — v1 Design

Status: approved (brainstorm) · Date: 2026-06-16 · App: `wp-site-connector` (Imperal Cloud extension)

Reference: build everything against the local docs mirror at
`Imperal OS/Docs/imperal-docs/` (INDEX.md → "WP Site Connector — build cheat-sheet").

## 1. Summary

A Python extension for Imperal Cloud that lets a user connect one or more WordPress sites
(via URL + Application Password) and **read-only** browse their content and health — through a
panel dashboard and by asking Webbee in chat. No writes to site content in v1.

## 2. Goals / Non-goals

**v1 goals**
- Connect multiple WordPress sites using a site URL + WP **Application Password** (Basic auth over HTTPS).
- See a list of connected sites with a connectivity status badge.
- Read **posts, pages, media** per site.
- See **site health** (best-effort, read-only — see §11).
- Both surfaces: a master-detail **panel** dashboard and **chat-functions** so Webbee can answer "show recent posts on {site}".
- Disconnect a site (with confirmation).

**Non-goals (v1)** — create/edit/delete site content; publishing pipeline; bulk multi-site
operations; a WordPress-side companion plugin; webhooks / scheduled polling (sync is on demand,
i.e. on panel open / on request).

## 3. Architecture & files

Plain Imperal extension (no HTTP server; handlers run in-process on the ICNLI Worker). Python 3.11+.

```
Apps/WP Site Connector/src/
├── main.py            # thin CLI loader (imports app)
├── app.py             # Extension(...) + ChatExtension(...)
├── handlers_read.py   # list_sites, list_posts, list_pages, list_media, get_site_health
├── handlers_connect.py# connect_site (form action), forget_site (chat destructive)
├── panels.py          # dashboard panel(s) + connection form
├── skeleton.py        # ambient "N sites connected" probe
├── models.py          # Pydantic param models + SDL entities
├── wp_client.py       # WP REST helpers over ctx.http (auth header, error mapping)
├── icon.svg           # valid viewBox, ≤100 KB
└── (imperal.json)     # generated by `imperal build`
```

`Extension("wp-site-connector", version=..., display_name="WP Site Connector",
description=<≥40 chars, concrete verbs/nouns — drives intent classification>, icon="icon.svg",
actions_explicit=True, capabilities=[...])`. `ChatExtension(ext, tool_name="wp-site-connector", description=...)`.

## 4. Data model

**`ctx.store` collection `sites`** (permanent, scoped by `ctx.user.imperal_id`), one record per site:
```
{ id, name, url, username, status: "connected"|"error", last_checked, last_error?, health: {...} }
```
`id` is a stable slug derived from the URL host (e.g. `example-com`).

**Credentials — one JSON secret (decision #1).** A single `@ext.secret("wp_credentials",
write_mode="both", max_bytes=16384, required=False)` holds a JSON map `{ site_id: app_password }`,
written via `ctx.secrets.set(...)` from the connection-form handler and read via
`ctx.secrets.get(...)`. All KMS-encrypted. `max_bytes` 16384 comfortably holds dozens of sites.
On connect: read map → add entry → set. On forget: read map → delete entry → set.

## 5. Security (non-negotiable)

- **The Application Password must never be an argument of an LLM-visible `@chat.function`.** It is
  captured only through the panel **connection form's direct submit action** (`ui.Form` → form
  handler), which bypasses the intent classifier. The LLM-facing tools are read-only + `forget_site`
  (which takes only `site_id`).
- Read credentials only via `ctx.secrets`; never from `ctx.config` / `os.environ`. Never log, cache,
  or return a password. Never put it in `ActionResult.data` (which feeds the fact-ledger).
- All store/secret access scoped by `ctx.user.imperal_id` (multi-tenant, fail-closed).
- Outbound only to the user's declared site host(s) via `ctx.http`; declare external-network intent
  in the listing description (triggers human review at publish).
- Send credentials as `Authorization: Basic base64(username:app_password)` over HTTPS only; refuse
  non-HTTPS site URLs.

## 6. Read tools (chat-functions)

All `action_type="read"`, each with an SDL `data_model=`. Each calls `wp_client` → `ctx.http.get(...)`
(timeout ≤ 180 s), checks `status_code` before `.json()`, maps 401/404/429/5xx to clean messages.

| Tool | Params | Returns (SDL) |
|---|---|---|
| `list_sites` | — | `sdl.EntityList[Site]` |
| `list_posts` | `site_id` (required, "from a previous list — never invent"), `limit?`, `search?` | `sdl.EntityList[Post]` |
| `list_pages` | `site_id`, `limit?`, `search?` | `sdl.EntityList[Page]` |
| `list_media` | `site_id`, `limit?` | `sdl.EntityList[MediaItem]` |
| `get_site_health` | `site_id` | `SiteHealth` entity |

SDL entities (`models.py`) subclass `sdl.Entity` (required `id`/`title`/`kind`) + relevant facets,
returned via `ActionResult.success(entity_or_list, summary=...)`.

WordPress endpoints: `GET /wp-json/wp/v2/posts|pages|media?per_page=&page=&search=&_fields=...`,
`GET /wp-json/wp/v2/users/me` (validation), `GET /wp-json/` (site name/description/namespaces).

## 7. Connection flow (panel form only)

1. Panel shows `ui.Form` (action → `connect_site` form handler) with `ui.Input(type="url")` for the
   site URL, `ui.Input` for username, **`ui.Password`** for the Application Password, `ui.Tooltip`
   on every label (manager-friendly text), submit `ui.Button`.
2. `connect_site` (form action, **not** LLM-exposed; `action_type="write"`): validate URL is HTTPS →
   `GET {url}/wp-json/wp/v2/users/me` with Basic auth → on 200, build site record, write to `ctx.store`,
   merge password into the `wp_credentials` secret map; on failure return a clear `ActionResult.error`.
3. `ActionResult(..., refresh_panels=["dashboard"])` so the list updates.

## 8. Disconnect flow

`forget_site(site_id)` — `@chat.function(action_type="destructive")` → the web-kernel shows the
**KAV confirmation card** automatically (byte-identical args). On confirm: remove the store record and
delete the entry from the `wp_credentials` map. No password in args.

## 9. Panels (UI)

Single master-detail dashboard panel (`@ext.panel("dashboard", slot="left")` for the site list +
`@ext.panel("detail", slot="center", center_overlay=False)` for content). Must return `ui.Empty()`
when nothing selected (never `None`); no host tab strip — branch on an `active_tab` kwarg.
- **Left:** `ui.List` of `ui.ListItem` (site name + `ui.Badge` status connected/error), `on_click`
  → `ui.Call` the detail panel with `site_id`. A "+ Connect site" button opens the connection form.
- **Center:** site header + `ui.Tabs` (Posts / Pages / Media) each rendering a `ui.List`/`ui.DataTable`;
  a health `ui.Card` (see §11). Refresh after connect/forget via `refresh_panels`.

## 10. Skeleton

`@ext.skeleton` returning small ambient counts (`{"response": {"sites_connected": N,
"last_synced": ...}}`) so Webbee routes "show my WP posts" without a wasted call. No rich lists.

## 11. Health surface + degradation (decision #2, accepted)

Read-only via standard REST, **best-effort**:
- **Available now:** reachability (does `wp-json` respond), auth-ok (does `users/me` succeed),
  HTTPS/SSL validity (checked on our side), content counts (posts/pages/media totals via
  `X-WP-Total` headers), site name/description (`/wp-json/`).
- **Not available without a companion plugin / writes — shown but badged "requires companion
  plugin (vNext)":** available plugin/core update counts, PHP version. (Authenticated
  `/wp/v2/plugins` can list installed plugins but not "update available"; PHP version is not in
  core REST.)

`SiteHealth` entity carries the available fields and marks the rest as unavailable rather than failing.

## 12. Error handling

`ActionResult.error("user-safe message", retryable=...)` — `retryable=True` only for timeout/429/5xx.
Never `error(str(e))`; `await ctx.log(...)` the detail, return a stable message. Map: 401 → "credentials
rejected — reconnect the site"; 404 → "REST API not found — is this a WordPress site / is it enabled";
403 → "user lacks permission"; timeout → "site did not respond".

## 13. Manifest / capabilities / validators

`imperal.json` generated by `imperal build`. Declare `capabilities` (incl. secrets `wp_credentials`),
tools with correct `action_type`/`data_model`/`effects`, the panels, the skeleton. Must pass
validators V14–V24 + V31 with **0 ERRORs**. `description` ≥ 40 chars and concrete. Read tools need
`data_model=` (V23). At least one test per `@chat.function` using `MockContext` (assign a
`MockSecretStore` for the secret).

## 14. Success criteria (acceptance)

1. Connect a site via URL + Application Password → it appears in the list with a status badge.
2. Browse its posts, pages, media; see available health with vNext fields badged.
3. Ask Webbee "show recent posts on {site}" → returns them.
4. Disconnect requires the KAV confirmation; afterward the site and its stored password are gone.
5. No credential ever appears in an LLM-visible arg, a log, or `ActionResult.data`.
6. `imperal validate .` = 0 errors; ≥1 test per chat-function passes.

## 15. Open questions / vNext
